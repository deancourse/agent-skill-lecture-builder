# 從零生成：先用 Codex 做出第一版講義網頁
> 先不要急著追求完美。第一步的目標，是用最短時間做出一個能被看見、能被討論、能被修正的版本。

## 這堂課要解的問題

### 🎯 今天不是在比哪個 AI 最強
- 重點不是單一工具有多猛，而是培養多個 AI 工具切換的能力
- 當一個工具紅起來後，往往也會迎來衰退；會切換工具的人更有主動權
- 與其被平台綁死，不如思考如何把自己的工作流設計成可延續的框架

> **工具會更替，方法才會留下**
> 我想分享的不是某一款工具的神話，而是如何在工具快速更迭的時代，保留自己的節奏、判斷與產能。

## 在終端機開始操作

### 🧰 先準備工作區
- 打開自己熟悉的程式碼編輯器，例如 VSCode、Cursor、Antigravity
- 建立一個空資料夾給 AI Agent 工作，避免內容混雜
- 這次重點不是依賴編輯器內建 AI，而是觀察專案檔案如何被建立與演進

### ✅ 確認 Node.js 環境

```prompt [label="確認版本"]
node -v
```

- 如果沒有顯示版號，先安裝 Node.js
- 建議透過 NVM 管理版本，後續切換與更新會比較輕鬆

```prompt [label="安裝 NVM（macOS）"]
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
```

```prompt [label="安裝最新 LTS"]
nvm install --lts
node -v
```

### 🚀 安裝 Codex

```prompt [label="全域安裝"]
npm i -g @openai/codex
```

- 可使用 ChatGPT 帳號登入，這是最推薦的方案
- 若在沒有瀏覽器的環境，也可以使用裝置驗證碼登入
- 也能改用 API Key，但要自行留意額度與費用控管

### 💬 第一個生成網頁的 Prompt

```prompt [label="生成純 HTML 講義頁"]
請扮演熟悉 AI 主題授課的講師，以「Agent Skills」為主題設計講義
講義用純 html 網頁（CSS/JS 都放裡面）呈現
日系風格的設計感，部分文字與背景可以搭配適當的漸層
開頭會根據內容呈現標題，然後下面有發人深省的標語
會根據不同層級與意圖呈現對應的區塊（ex: 大小標題、陳述、列點、流程）
最後放上社群連結鼓勵大家訂閱（FB/YouTube/Medium）
側邊欄會有對應的目錄，方便跳轉到對應位置
```

[flow]
1. 先定義目標：只生成單一 HTML，方便部署與攜帶
2. 再描述風格：指定視覺方向與版面氣質
3. 最後交代架構：首頁、內容區塊、社群收尾、側邊目錄
[/flow]

### ⚠️ 第一版常見痛點
- 與 AI 對話微調內容時，隨機性太高
- 直接修改 HTML，維護成本又太重
- 產得很快，不代表後續好改

> **快，不等於可用**
> 如果後續每次修改都要重新賭一次輸出結果，那麼再漂亮的第一版，也很難真的進入日常工作流。

# 建立框架：把隨機輸出轉成可維護的 Markdown
> 想對內容有掌控力，就不能把知識鎖在 HTML 成品裡。真正該維護的，應該是結構化內容。

## 換一種思路

### 🏗️ 框架的核心概念
- 不要讓 AI 每次都從零生成整個成果
- 先建立一套穩定模板，再替換不同內容
- 重點從「一次性生成」轉向「可重複使用」

### ✍️ 逆向設計 Prompt

```prompt [label="把 HTML 轉成框架"]
我希望剛剛生成出來的網頁，是可以透過 Markdown 格式編輯內容的
請幫我將網頁結構拆解（列點、步驟、圖片、表格、註解）、分層次（大標、中標、小標），並以常見的 Markdown 語法來設計規則
最後建立一個腳本，我只要編輯 Makdown 文件，網頁的 HTML 就會及時改變
操作方式寫在 README.md 文件中，並建立 package.json
```

[summary]
- 🧱 **內容格式化** | 把 HTML 成品逆向拆回可編輯的 Markdown 結構
- ⚙️ **腳本自動化** | 讓內容調整後能重新編譯成穩定頁面
- 📘 **文件化** | 把操作方式寫進 README，降低未來維護成本
[/summary]

### 🔁 建立後的操作方式

```prompt [label="啟動本機預覽"]
npm run dev
```

- 開啟對應網址後，可以看到頁面已由 Markdown 成功渲染
- 只要修改內容檔案，網頁就會同步更新
- 這時候我們就把維護重心從 HTML 轉回內容本身

## 為什麼 Markdown 很重要

[tags]
- [green] 內容容易調整，細節可手動精修
- [blue] 格式穩定，便於建立團隊共識
- [purple] 能與腳本、模板、設定檔分工合作
- [orange] 若結構沒設計好，後續擴充仍然會痛苦
[/tags]

> **你不是在做一個頁面，而是在設計一種生產方式**
> 一旦內容、配置、樣式、設定被拆開管理，後續每一次修改都會輕鬆很多。

# 封裝成 Skill：讓整個流程可以被反覆使用
> 當工作流被驗證有效，就該把它包成技能，讓未來不必重來一次。

## 建立 Agent Skill

### 🧠 Skill 要描述什麼

```prompt [label="建立課程頁 Skill"]
根據下面需求在「.agents」資料夾建立 Skill：
- 參考模板：根據剛剛腳本設計的規格，建立明確的 Markdown 使用規範
- 觸發時機：使用者提供初步想法想建立網頁時，會觸發這個 Skill
- 執行順序：這個 Skill 會先深入研究輸入的主題，再根據規範設計對應的 Markdown 文件；最後把 Markdown 文件透過腳本生成 html
- 特別提醒：預計會有多個主題，所以觸發時，請先建立新的資料夾，裡面存放新生成的 Markdown 與 HTML 檔案；並且建立獨立的 scripts 與 references 資料夾，確保可單獨執行
```

[flow]
1. 定義觸發時機：什麼情況下應該啟用這個 Skill
2. 定義執行步驟：研究主題、整理 Markdown、編譯成 HTML
3. 定義輸出結構：每個主題都建立獨立資料夾方便管理
[/flow]

### 📁 Skill 建好後會包含什麼
- `SKILL.md`：定義名稱、用途與執行流程
- `reference/`：放 Markdown 格式規範與範例
- `scripts/`：放 build、preview、OG 圖等腳本

### 🧪 驗證 Skill 是否可用

```prompt [label="驗證 Skill"]
我想要建立深度介紹「ChatGPT 使用技巧」的課程講義網頁。
```

- 若有成功觸發 Skill，AI 會自動走完整個建立流程
- 最終會產出 Markdown 與 HTML，不再需要每次手工重搭頁面

# 走向產品：自由度高，也代表你要承擔維護責任
> 自建框架很自由，但也表示你是這套工具目前最懂、也最需要持續優化的人。

## 真正使用後才會看見的需求

### 🪟 講課模式不是一般閱讀模式
- 側邊欄需要可收合，避免佔據投影空間
- 講者需要知道目前授課進度，最好有進度條
- 指令區塊需要一鍵複製，方便單手操作

### 🌗 場地不同，顯示需求也不同
- 淺色模式適合一般閱讀
- 深色模式更適合關燈授課的場地
- 主題切換不只是美觀，而是實際使用情境需求

### 🧾 PDF 不能直接拿網頁硬印
- 若要提供講義下載，列印版面需要獨立設計
- 直接用瀏覽器預設列印，通常不會有好效果

> **主控權的代價，就是要面對複雜度**
> 你想要更貼近自己需求的工具，就要願意在真實使用後持續修正細節。

## 成果展示背後的設計思維

### 📦 共用資訊與客製資訊要分開
- 講者資料、社群連結、預設 OG，可放進 `global.yaml`
- 每一篇課程頁特有的標題、摘要、引言，放進 `config.yaml`
- 內容本體則放在 Markdown，讓更新頻率最高的資訊最好改

### 🔎 專業度會反映在細節
- 若頁面打算公開分享，就要留意 Open Graph 設定
- SEO 標題、描述、縮圖沒有設好，分享出去的第一印象就會打折
- 架構合理與否，仍需要人的判斷，而不是全交給 AI

[summary]
- 🎨 **畫面** | 側邊欄、深色模式、講課進度與 PDF 版面都要為真實情境設計
- 🗂️ **架構** | Markdown、課程設定、全域設定拆分，維護才會輕鬆
- 🧭 **判斷** | AI 能加速實作，但方向感與品質標準仍然掌握在人手上
[/summary]

# 結語：Agent Skill 的價值，是把工作流變成系統
> AI 最厲害的地方，不只是幫你省下一次工作的時間，而是讓你有機會把自己的方法論變成可持續優化的工具。

## 今天做了哪三件事

[flow]
1. 用 Codex 從零生成第一版講義網頁
2. 把隨機輸出整理成可維護的 Markdown 框架
3. 把流程封裝成 Agent Skill，讓未來可以反覆使用
[/flow]

### 💡 最值得帶走的觀念
- 不要只追求「做出來」，更要思考「之後怎麼改」
- 不要只把 AI 當生產工具，也要把它當成框架設計的協作者
- 當你開始設計自己的系統，Vibe Coding 才真的從玩具走向產品

> **知道自己要什麼，然後動手做就對了**
> 從這一刻開始，你不只是使用別人的工具，而是有能力慢慢把工具設計成更符合自己需求的樣子。
